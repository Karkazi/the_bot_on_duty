# Предложение: Автоматический сбор данных для обучения AI-ассистента по диагностике сбоев

## 🎯 Суть предложения

Добавить в бота функционал автоматического сбора и экспорта данных о технических сбоях в Excel файл для обучения нейронной сети. В будущем это позволит создать AI-ассистента, который будет анализировать описание сбоя и автоматически предлагать вероятную причину и решение.

---

## 💡 Проблема, которую решаем

**Текущая ситуация:**
- При возникновении сбоя инженер тратит время на анализ проблемы и поиск решения
- Нет накопленной базы знаний о типичных сбоях и их решениях
- Опыт решения проблем не систематизирован и не используется для ускорения диагностики
- Каждый раз приходится "изобретать велосипед" при похожих проблемах

**Решение:**
- Автоматически собирать данные о всех решенных сбоях
- Накопить базу знаний для обучения AI-модели
- В будущем получить AI-ассистента, который будет предлагать решения на основе накопленного опыта

---

## ✨ Выгоды и преимущества

### 🚀 Немедленные выгоды

1. **Автоматизация рутины**
   - Не нужно вручную собирать данные о сбоях
   - Все данные собираются автоматически при закрытии сбоя
   - Готовый Excel файл для анализа и обучения

2. **Сохранение знаний**
   - Вся информация о решенных сбоях сохраняется в структурированном виде
   - История решений не теряется
   - Можно анализировать паттерны и тренды

3. **База для обучения**
   - Готовая структурированная выборка для обучения нейронной сети
   - Данные уже очищены и валидированы
   - Можно сразу использовать для обучения модели

### 🎯 Долгосрочные выгоды

1. **AI-ассистент для диагностики**
   - После обучения модели получим помощника, который:
     - Анализирует описание сбоя
     - Предлагает вероятную причину
     - Предлагает возможные решения
   - Ускорение диагностики в 2-3 раза
   - Снижение времени на решение типичных проблем

2. **Улучшение качества обслуживания**
   - Быстрее находить решение для повторяющихся проблем
   - Меньше времени простоя сервисов
   - Повышение удовлетворенности пользователей

3. **Накопление экспертизы**
   - Систематизация знаний команды
   - База знаний о типичных проблемах и решениях
   - Обучение новых сотрудников на основе исторических данных

## 📊 Что будет собираться

### Основные данные о сбое

| Данные | Зачем нужно | Источник |
|--------|-------------|----------|
| **Описание сбоя** | Основной вход для AI - что анализировать | Бот (пользователь ввел) |
| **Сервис** | Категоризация, контекст | Бот (выбрал пользователь) |
| **Причина сбоя** | Целевое значение для обучения AI | Jira (кастомное поле) |
| **Решение** | Целевое значение для обучения AI | Jira (кастомное поле) |
| **Время до решения** | Метрика эффективности | Бот + Jira |
| **История обсуждения** | Дополнительный контекст | SCM канал (Telegram) |

### Полная структура данных

Файл будет содержать 15 колонок с информацией:
- Идентификаторы (ID сбоя, Jira ключ, SCM topic ID)
- Временные метки (создание, устранение, длительность)
- Описательные данные (описание, сервис, уровень проблемы)
- **Целевые данные для обучения** (причина, решение)
- Контекст (история обсуждения, метаданные)

## 🔄 Как это будет работать

### Процесс сбора данных

1. **Создание сбоя** (уже работает)
   - Пользователь создает сбой через бота
   - Данные сохраняются в боте

2. **Решение сбоя** (уже работает)
   - Сбой решается, задача в Jira закрывается
   - В Jira заполняются поля "Причина" и "Решение"

3. **Автоматический сбор** (новый функционал)
   - При закрытии сбоя бот автоматически:
     - Получает данные из Jira (включая причину и решение)
     - Собирает историю обсуждения из SCM канала
     - Сохраняет все данные в историю закрытых сбоев

4. **Экспорт в Excel** (новый функционал)
   - По запросу администратора генерируется Excel файл
   - Все данные валидируются и очищаются
   - Файл готов для обучения нейронной сети

### Пример данных

```
Описание сбоя: "Не работает авторизация на сайте. Пользователи не могут войти в личный кабинет."
Сервис: Сайт Petrovich.ru
Причина сбоя: "Превышено время ожидания ответа от сервиса авторизации. Таймаут установлен на 5 секунд, но сервис отвечает за 8-10 секунд."
Решение: "Увеличен таймаут запросов до 15 секунд. Добавлен retry механизм с экспоненциальной задержкой."
Время до решения: 85 минут
```

## 🛠️ Техническая реализация

### Что нужно добавить в бота

1. **Расширение работы с Jira API**
   - Получать не только статус, но и полные данные задачи
   - Извлекать кастомные поля "Причина" и "Решение"
   - Кэшировать данные для оптимизации

2. **Сохранение истории**
   - При закрытии сбоя сохранять его в отдельный файл/БД
   - Сохранять все данные, включая причину и решение
   - Добавить метаданные (дата закрытия, способ закрытия)

3. **Сбор данных из SCM**
   - Получать историю сообщений из темы SCM канала
   - Форматировать сообщения в единый текст
   - Обрабатывать пагинацию (если сообщений много)

4. **Генерация Excel**
   - Собрать данные из всех источников
   - Валидировать и очистить данные
   - Создать Excel файл с нужной структурой
   - Добавить метаданные (версия, дата создания)

### Требования

- **Минимальные изменения**: Используем существующие интеграции (Jira, Telegram)
- **Производительность**: Кэширование данных, batch-обработка
- **Безопасность**: Данные не покидают инфраструктуру компании
- **Масштабируемость**: Поддержка до 10,000 записей в файле

## ⚠️ Риски и ограничения

### Потенциальные риски

1. **Качество данных в Jira**
   - Если поля "Причина" и "Решение" не заполняются - данные неполные
   - **Решение**: Добавить валидацию и уведомления о неполных данных

2. **Производительность API**
   - Jira и Telegram API могут ограничивать запросы
   - **Решение**: Кэширование, batch-обработка, rate limiting

3. **Конфиденциальность**
   - Excel файл содержит чувствительные данные
   - **Решение**: Безопасное хранилище, контроль доступа

### Ограничения

- Нужно, чтобы в Jira заполнялись поля "Причина" и "Решение"
- Telegram API может ограничивать количество сообщений из истории
- Размер файла ограничен (рекомендуется до 10,000 строк)
