# План: чаты по сбоям в MAX (ALARM_MAIN + FA-****)

## Идея в двух словах

Пока в MAX нет тем в чатах, используем **один общий чат (ALARM_MAIN)** для уведомлений о сбоях и **отдельный чат на каждый сбой (FA-****)** для обсуждения. После решения сбой закрывается, история сохраняется, чат FA-**** удаляется. Дополнительно: мост сообщений между **темой в Telegram (SCM-канал)** и **временным чатом FA-**** в MAX**, а также модерация ALARM_MAIN (только бот и админы).

---

## 1. Чаты и роли

| Чат | Назначение | Кто в чате |
|-----|------------|------------|
| **ALARM_MAIN** | Единственный «лобный» чат по сбоям. Только посты бота (новый сбой + ссылка на чат FA-****) и при необходимости сообщения админов. Без общего флуда. | Все, кто решает сбои; бот (с правом удалять чужие сообщения). |
| **FA-1234** (на каждый сбой) | Обсуждение конкретного сбоя в MAX. Вся переписка по FA-1234 идёт здесь. Параллельно та же переписка дублируется в **тему** в Telegram (см. п. 4). После закрытия сбоя — история сохраняется, чат удаляется. | Назначенные/заинтересованные; бот (создатель, позже — экспорт и удаление). |
| **Тема в TG (SCM_CHANNEL_ID)** | Уже реализовано в боте: при создании сбоя создаётся тема в канале SCM. Одна тема = один сбой (FA-XXXX). Мост синхронизирует сообщения между этой темой и чатом FA-XXXX в MAX. | Участники канала; бот (создаёт тему, пересылает сообщения MAX ↔ TG). |

---

## 2. Сценарии

### 2.1 Заведение сбоя (создание чата FA-****)

1. Пользователь заводит сбой в боте (Telegram или MAX) как сейчас → создаётся задача в Jira (FA-XXXX).
2. **Новое:** бот в MAX создаёт чат с названием `FA-XXXX` (или «Сбой FA-XXXX»).
3. Бот добавляет в этот чат: инициатора сбоя и при необходимости список «решателей» (из конфига / Jira / ролей).
4. Бот отправляет в **ALARM_MAIN** одно сообщение вида:
   - «Новый сбой: FA-XXXX. [краткое описание]. Обсуждение: [ссылка на чат FA-XXXX].»
5. Дальнейшее обсуждение — только в чате FA-XXXX.

### 2.2 Обсуждение в чате FA-**** и в теме TG

- Участники пишут либо в чат FA-XXXX в MAX, либо в **тему** по этому сбою в Telegram (канал `SCM_CHANNEL_ID`).
- **Мост MAX ↔ TG:** сообщения из чата FA-XXXX в MAX пересылаются в соответствующую **тему** в TG (одна тема на сбой — она уже создаётся ботом при заведении сбоя). Сообщения из темы в TG пересылаются в чат FA-XXXX в MAX. Таким образом обсуждение идёт и в теме в TG, и во временном чате FA-**** в MAX — синхронно.

### 2.3 Закрытие сбоя и архивация

1. Сбой закрывают (как сейчас: кнопка «Сбой устранён» / закрытие в Jira и т.п.).
2. **Новое:**
   - Бот получает историю чата FA-XXXX из MAX API.
   - Сохраняет историю:
     - **Вариант A:** в файл Excel (дата, автор, текст, вложения — ссылки/описание);
     - **Вариант B:** комментарием в задачу Jira FA-XXXX (например, один большой комментарий с логом переписки);
     - **Вариант C:** оба варианта (Excel для отчётов + комментарий в Jira для контекста).
3. Бот удаляет чат FA-XXXX в MAX (если API MAX позволяет удаление чата ботом).
4. В ALARM_MAIN можно опционально отправить короткое сообщение: «Сбой FA-XXXX закрыт, история сохранена».

---

## 3. ALARM_MAIN: только бот и админы

- **Цель:** в ALARM_MAIN не должно быть посторонних обсуждений — только уведомления бота и при необходимости сообщения админов.
- **Механика:**
  - Боту выдать права в чате ALARM_MAIN так, чтобы он мог **удалять сообщения** (и при необходимости — только от не-админов).
  - При появлении нового сообщения в ALARM_MAIN бот проверяет: автор — сам бот или пользователь из списка админов MAX?
    - Если да — оставить.
    - Если нет — удалить сообщение (и при желании отправить автору в ЛС короткое напоминание: «Обсуждение сбоев ведётся в чате по ссылке из сообщения бота»).
- **Реализация:** обработчик событий «новое сообщение в ALARM_MAIN» (по `chat_id` = ALARM_MAIN); проверка автора; вызов API удаления сообщения для чужих постов.

## 4. Мост сообщений: тема в TG (SCM) ↔ временный чат FA-**** в MAX

### 4.1 Уже реализовано в боте

- При заведении сбоя (и при выборе «Завести тему в SCM») бот создаёт **тему форума** в Telegram-канале с ID из конфига **`SCM_CHANNEL_ID`**.
- Название темы связано с сбоем (FA-XXXX, описание). **ID темы** (`scm_topic_id` / `message_thread_id`) сохраняется в данных сбоя (`bot_state.active_alarms[alarm_id]["scm_topic_id"]`).
- Есть отправка сообщений в тему: `channel_service.send_to_scm_topic(channel_id, topic_id, text)`.
- Таким образом, **одна тема в TG уже соответствует одному сбою (FA-XXXX)**. Для моста нужно связать эту тему с **временным чатом FA-XXXX в MAX**.

### 4.2 Связка: тема в TG ↔ чат FA-**** в MAX

- **Пара «тема TG — чат MAX»:** для сбоя FA-XXXX есть ровно одна тема в канале `SCM_CHANNEL_ID` (topic_id = `scm_topic_id`) и один временный чат в MAX с названием/привязкой FA-XXXX.
- Обмен сообщениями идёт **только** между этой темой и этим чатом (не между разными чатами/темами).

### 4.3 Потоки сообщений

- **MAX → TG:** при новом сообщении в чате FA-XXXX в MAX бот пересылает его в **тему** с `message_thread_id = scm_topic_id` в канале `SCM_CHANNEL_ID` (с подписью вида «[MAX, Имя]: текст»).
- **TG → MAX:** при новом сообщении в этой теме в TG (от пользователя, не от бота) бот отправляет его в чат FA-XXXX в MAX (от имени бота с подписью «[TG, Имя]: текст» или «Имя (TG): текст»).

Так участники могут писать и в теме в Telegram, и во временном чате в MAX — переписка синхронизируется между темой в TG и чатом FA-**** в MAX.

### 4.4 Сложности

- Маппинг пользователей MAX ↔ Telegram (user_id / username / явная привязка в конфиге).
- Определение «сообщение в теме по сбою FA-XXXX» в TG: по `chat_id = SCM_CHANNEL_ID` и `message_thread_id = scm_topic_id` для нужного alarm_id.
- При архивации: объединять историю из MAX (чат FA-XXXX) и из TG (тема) в одну хронологию по времени.

## 5. Зависимости от API MAX (итоги исследования)

По библиотеке **maxapi** и документации **dev.max.ru** (platform-api.max.ru):

| Действие | Результат |
|----------|-----------|
| **Создание чата** | В API **нет** метода «создать чат». Бот может только добавлять участников в уже существующий чат (`POST /chats/-chatId-/members`), получать чат по ссылке (`GET /chats/-chatLink-`) и по ID. Чаты FA-**** нужно создавать вручную в MAX или ждать появления метода в API. |
| **Получение истории** | **Есть**: `GET /messages` с параметрами `chat_id`, `count`, `from`, `to` (в maxapi: `bot.get_messages(chat_id=..., count=...)`). |
| **Удаление чата** | **Есть**: `DELETE /chats/-chatId-` (в maxapi: `bot.delete_chat(chat_id)`). |
| **Удаление сообщения** | **Есть**: `DELETE /messages` (в maxapi: `bot.delete_message(...)`). Модерация ALARM_MAIN реализована. |
| **Ссылка на чат** | **Есть**: `GET /chats/-chatLink-` (получение чата по ссылке). Формат ссылки для вставки в ALARM_MAIN уточняется по документации MAX. |

**Вывод:** этапы 5 (архивация) и 6 (мост) реализуемы при условии, что у сбоя есть `max_chat_id` (чат FA-XXXX создан вручную или позже через API). Этап 3 (автосоздание чата FA-XXXX) в текущем API недоступен — возможны ручное создание чата и привязка его к сбою (например, через `max_chat_id` в данных сбоя).

### Изменение имени (названия) чата

| Вопрос | Ответ |
|--------|--------|
| **Может ли бот менять имя чата?** | **Да.** В API MAX есть метод **`PATCH /chats/-chatId-`** (в maxapi — `bot.edit_chat(chat_id, title="...")`). |
| **Параметры** | `title` — новое название чата, **1–200 символов**. Дополнительно: `icon` (иконка), `pin` (id закреплённого сообщения), `notify` (уведомления). |
| **Где в maxapi** | `maxapi.bot.Bot.edit_chat(chat_id, title="Новое название", ...)` → класс `EditChat`, документация: https://dev.max.ru/docs-api/methods/PATCH/chats/-chatId- |
| **Применение** | Например, при закрытии сбоя можно переименовать чат FA-XXXX в «✅ FA-XXXX устранён» перед архивацией или оставить для истории. Либо задать название при создании чата вручную, а потом при закрытии обновить через бота. |

## 6. Конфиг и окружение

- **ALARM_MAIN** — один раз завести чат в MAX, положить его `chat_id` в конфиг (например `MAX_ALARM_MAIN_CHAT_ID` в `.env`).
- **SCM_CHANNEL_ID** (Telegram) — уже используется: канал с темами для сбоев; при создании сбоя бот создаёт тему и сохраняет `scm_topic_id` в данных сбоя. Мост MAX ↔ TG опирается на этот канал и эти topic_id.
- Список админов MAX уже есть (`MAX_ADMIN_IDS`); их не удаляем в ALARM_MAIN.
- При необходимости: отдельный конфиг для «чатов сбоев» в MAX (префикс названия, кто добавляется по умолчанию, лимит участников).

## 7. Этапы реализации (предварительно)

1. **Исследование API MAX** — создание чата, история, удаление чата/сообщения, ссылки на чат.
2. **ALARM_MAIN: уведомления и модерация** — ✅ реализовано:
   - В конфиг добавлены `MAX_ALARM_MAIN_CHAT_ID` и опционально `MAX_BOT_USER_ID`.
   - При создании сбоя бот отправляет в ALARM_MAIN сообщение: «Новый сбой: FA-XXXX», описание, Jira (если есть), «Обсуждение: тема в канале SCM».
   - Модерация: в ALARM_MAIN удаляются все сообщения, автором которых не является бот и не пользователь из `MAX_ADMIN_IDS`; при необходимости задают `MAX_BOT_USER_ID`, чтобы не удалять посты бота.
3. **Создание чата FA-XXXX** — в API MAX метода создания чата нет. Чаты создают вручную в MAX; затем можно привязать чат к сбою, задав у сбоя поле `max_chat_id` (вручную в `data/state.json` или через будущую команду/админку). После привязки работают архивация (п. 5) и мост (п. 6).
4. ~~**Модерация ALARM_MAIN**~~ — см. п. 2.
5. **Архивация** — ✅ реализовано: при закрытии сбоя, если у него задан `max_chat_id`, бот получает историю чата из MAX (`GET /messages`), сохраняет в `data/archive/alarms/FA-XXXX_YYYYMMDD_HHMM.txt` и удаляет чат (`DELETE /chats/-chatId-`). Поле `max_chat_id` в данных сбоя задаётся вручную (чат в MAX создаётся вручную) или позже через API.
6. **Мост MAX ↔ TG** — ✅ реализовано: сообщения из чата MAX (где `max_chat_id` сбоя) пересылаются в тему SCM в TG (`[MAX, Имя]: текст`); сообщения из темы SCM в TG пересылаются в чат MAX (`[TG, Имя]: текст`). Работает при заданных у сбоя `max_chat_id` и `scm_topic_id`.

Можно дробить этапы мельче (например, сначала только уведомления в ALARM_MAIN без отдельных чатов, потом добавить чаты и архивацию).

## 8. Риски и упрощения

- **API MAX не даёт создавать/удалять чаты** — тогда оставляем только ALARM_MAIN и одну общую переписку (или несколько чатов, созданных вручную), без автоматического «чата на сбой».
- **Нет доступа к истории чата** — архивация только вручную (копипаст) или только метаданные (кто был в чате, время создания/закрытия).
- **Мост TG ↔ MAX** — пересылка между темой в SCM-канале и чатом FA-****; можно отложить и сначала запустить сценарий только в MAX (ALARM_MAIN + чаты FA-**** + архивация), тогда темы в TG работают как сейчас (без синхронизации с MAX).

Если нужно, следующий шаг — детализация одного из этапов (например, только ALARM_MAIN и модерация) или проверка maxapi/документации MAX по пункту 5.
