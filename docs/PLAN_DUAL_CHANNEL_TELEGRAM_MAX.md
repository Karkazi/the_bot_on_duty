# План: управление ботом из Telegram и MAX (вариант А)

**Цель:** один и тот же функционал бота доступен и в Telegram, и в MAX. Логика одна (core), каналы два (Telegram + MAX, в MAX — **polling**).

---

## 1. Общая схема

```
┌─────────────────┐     ┌─────────────────┐
│   Telegram      │     │   MAX           │
│   (aiogram)     │     │   (polling)     │
│   handlers      │     │   handlers      │
└────────┬────────┘     └────────┬────────┘
         │                       │
         │    вызывают           │
         ▼                       ▼
┌─────────────────────────────────────────┐
│              CORE (общая логика)         │
│  • Действия: создать сбой, продлить,    │
│    остановить, список, помощь            │
│  • Состояние: bot_state + общий FSM     │
│  • Сервисы: Jira, SimpleOne, каналы     │
└─────────────────────────────────────────┘
         │                       │
         ▼                       ▼
┌─────────────────┐     ┌─────────────────┐
│  Ответ в TG     │     │  Ответ в MAX     │
│  (aiogram)      │     │  (max api)       │
└─────────────────┘     └─────────────────┘
```

---

## 2. Что уже есть и что нужно добавить

| Компонент | Сейчас | Нужно для варианта А |
|-----------|--------|----------------------|
| Логика (создать сбой, продлить, остановить и т.д.) | В handlers (Telegram) | Вынести в **core** (отдельный слой), вызывать из TG и MAX |
| Состояние (bot_state) | Есть, общее | Оставить общим; FSM по шагам диалога — сделать канал-независимым |
| Отправка в каналы (TG + MAX) | Уже дублируем уведомления в ALARM | Оставить как есть |
| MAX | Только отправка в канал (MaxService) | Добавить **вход**: polling, команды, кнопки, ответы пользователю |
| Запуск | Один процесс: aiogram polling | Один процесс: **два цикла** (aiogram + MAX polling) или одна event loop с двумя задачами |

## 3. Пошаговый план

### Этап 0. Подготовка (без ломки текущего кода)

**Шаг 0.1.** Зафиксировать список «действий» бота (то, что должно быть доступно из обоих каналов):

- `/start`, `/help`
- «Сообщить» → сбой / работы / обычное сообщение
- Создание сбоя (описание, сервис, время, Jira, SCM, Петлокал, подтверждение)
- Создание работ (описание, время, сервисы, Петлокал, подтверждение)
- Обычное сообщение (текст, Петлокал, подтверждение)
- «Управлять» → список сбоев/работ → остановить / продлить
- «Текущие события» → список активных сбоев и работ
- Напоминания (продлить/остановить по кнопке из напоминания)

**Шаг 0.2.** Решить, что в первой итерации делаем в MAX (минимум):

- Рекомендация для v1: только **чтение** и **простые команды**: `/start`, `/help`, «Текущие события», «Остановить по ID» / «Продлить по ID» (текстом). Создание сбоев/работ с полным сценарием (кнопки, шаги) — во вторую очередь.

**Шаг 0.3.** Добавить в `.env` и конфиг параметры для MAX как канала управления (если ещё нет):

- `MAX_BOT_TOKEN` — уже есть
- `MAX_ADMIN_IDS` — список user_id в MAX через запятую (например `123,456`); проверка в `config.is_max_admin(user_id)`
- `MAX_MANAGEMENT_ENABLED` — по умолчанию true при заданном `MAX_BOT_TOKEN`; иначе можно явно задать `0`/`false`
- При необходимости отдельный `MAX_API_URL` (или оставить default)

### Этап 1. Выделение core-действий

**Шаг 1.1.** Создать пакет (например `core/` или `use_cases/`) с функциями/классами, которые **не знают** про Telegram или MAX:

- `get_help_text()` — текст помощи (уже по сути есть в start_help).
- `get_active_events()` — форматированный список активных сбоев и работ (данные из `bot_state`).
- `stop_alarm(alarm_id)` / `stop_maintenance(work_id)` — остановка с обновлением state, отправкой в каналы (Telegram + MAX), Petlocal, SCM и т.д. Внутри вызывать существующую логику из `handlers.manage.stop` и сервисов.
- `extend_alarm(alarm_id, minutes)` / `extend_maintenance(work_id, minutes)` — продление, аналогично вынести из handlers в core.

Идея: текущие обработчики Telegram **постепенно** начинают вызывать эти функции вместо дублирования логики.

**Шаг 1.2.** Ввести «абстракцию ответа пользователю»:

- Небольшой слой «отправить сообщение пользователю»: на вход — `channel` (telegram | max), `user_id`, `chat_id`, `text`, опционально `keyboard`.
- Реализации: для Telegram — вызов `bot.send_message(chat_id, ...)`; для MAX — вызов `MaxService.send_message(chat_id, ...)` (уже есть отправка в канал, при необходимости расширить для личных чатов).
- Core-функции тогда получают «sender» (или callback) и шлют ответы через него, не зная канала.

**Шаг 1.3.** (Опционально для v1.) Вынести пошаговое состояние диалога (создание сбоя, выбор сервиса, времени и т.д.) в канал-независимый FSM:

- Ключ сессии: например `(channel, user_id)` или `("telegram", user_id)` / `("max", user_id)`.
- Состояние и данные хранить в `bot_state.user_sessions` или отдельном словаре, а не только в aiogram FSM.
- Это понадобится, когда будете переносить в MAX полный сценарий «Сообщить» с кнопками.

Для первой версии можно оставить FSM в Telegram и в MAX сделать только команды без многошагового диалога.

### Этап 2. Подключение MAX (polling)

**Шаг 2.1.** Выбрать способ работы с MAX:

- Использовать **maxapi** (как в боте Lupa): библиотека с polling, события `message_created`, `message_callback` и т.д.
- Установить зависимость: `maxapi` (проверить имя пакета и совместимость с Python).

**Шаг 2.2.** Создать модуль входа для MAX, например `handlers_max/` или `adapters/max/`:

- Инициализация бота MAX (токен из конфига).
- Обработчик входящих сообщений (текст команд).
- Обработчик callback от кнопок (если в v1 будут кнопки в MAX).

**Шаг 2.3.** Реализовать команды в MAX (текстом, без сложного FSM):

- `/start` — приветствие + краткая подсказка (вызвать `get_help_text()` или аналог, отправить ответ в тот же чат MAX).
- `/help` — полный текст помощи (core).
- Команда/фраза типа «список», «текущие», «события» → вызвать `get_active_events()` → отправить результат в чат MAX.
- «Остановить &lt;id&gt;» (например `Остановить FA-1234`) → парсить id → вызвать `stop_alarm(id)` или `stop_maintenance(id)` → отправить «Готово» или ошибку в MAX.
- «Продлить &lt;id&gt; &lt;минуты&gt;» → парсить id и минуты → вызвать core `extend_*` → ответ в MAX.

**Шаг 2.4.** Проверка прав в MAX:

- По `user_id` из события MAX проверять, что он входит в `MAX_ADMIN_IDS`.
- Если нет — ответить в MAX «Доступ запрещён» и не вызывать core.

**Шаг 2.5.** Запуск двух поллингов в одном процессе:

- В `main.py`: после запуска aiogram polling запустить вторую задачу — MAX polling (например `asyncio.create_task(run_max_polling())`).
- Либо запустить MAX polling в отдельном потоке с своей event loop — сложнее; проще одна loop, две корутины.
- При завершении бота (Ctrl+C, shutdown) корректно остановить и aiogram, и MAX polling.

### Этап 4. Расширение MAX до полного сценария (после v1)

**Шаг 4.1.** Перенести в MAX сценарий «Сообщить» (сбой/работы/обычное):

- Потребуется канал-независимый FSM (см. шаг 1.3).
- В MAX: по команде «сообщить» или кнопке переходить в состояние «выбор типа» → «сбой» → ввод описания → выбор сервиса (список кнопок в MAX) → время → Jira/SCM/Петлокал → подтверждение.
- Каждый шаг: показ клавиатуры в формате MAX, ожидание callback или текста, обновление FSM, вызов core (где уже есть логика создания сбоя/работы).

**Шаг 4.2.** Кнопки в MAX:

- Использовать maxapi (CallbackButton, InlineKeyboardBuilder — если есть аналог) для «Остановить», «Продлить», «Отмена», выбора сбоя/работы по ID.
- Callback payload кодировать единообразно, например `action=stop&type=alarm&id=FA-1234`, и в core разбирать один раз.

**Шаг 4.3.** «Управлять» в MAX:

- Команда «управлять» → список активных сбоев/работ с кнопками (каждый элемент — кнопка с payload `action=select&type=alarm&id=...`).
- По нажатию — показать «Остановить / Продлить / Отмена» для выбранного id и вызывать уже существующие core-функции.

## 4. Порядок реализации (кратко)

1. **Этап 0** — список действий, минимум для MAX v1, конфиг MAX_ADMIN_IDS.
2. **Этап 1** — core: `get_help_text`, `get_active_events`, `stop_*`, `extend_*` и слой «отправить ответ» (telegram/max).
3. **Этап 2** — MAX polling (maxapi), обработка `/start`, `/help`, «список», «остановить &lt;id&gt;», «продлить &lt;id&gt; &lt;мин&gt;», проверка MAX_ADMIN_IDS, запуск двух поллингов в main.
4. **Этап 5** — конфиг, логи, README.
5. Потом по необходимости — **Этап 3** (контекст), **Этап 4** (полный сценарий и кнопки в MAX).

## 5. Риски и упрощения

- **Конфликт FSM:** пока в MAX только простые команды без многошагового диалога, конфликта с aiogram FSM не будет. Один пользователь может одновременно писать и в TG, и в MAX — тогда два независимых «сеанса» (по одному на канал).
- **Формат кнопок:** в MAX кнопки и callback отличаются от Telegram — маппинг в общие action (stop, extend, select) делается в адаптере MAX, core получает уже готовые команды с параметрами.
- **Упрощение v1:** не переносить в MAX создание сбоев/работ с нуля; только просмотр списка и остановка/продление по текстовой команде с ID. Так быстрее получить рабочий вариант «по варианту А» с polling в MAX.

## 6. Итог

- **Вариант А** — одна логика (core), два входа (Telegram + MAX).
- В MAX используем **polling** (maxapi).
- Сначала делаем core и простые команды в MAX; затем при необходимости добавляем полный сценарий и кнопки в MAX по этому плану.

## 7. Реализовано (вариант А, этапы 1–2)

- **Конфиг:** `MAX_ADMIN_IDS`, `MAX_MANAGEMENT_ENABLED`, `is_max_admin()` в `config.py`.
- **Core:** пакет `core/` — `get_help_text()`, `get_active_events_text(view, page)`, `stop_alarm`, `stop_maintenance`, `extend_alarm`, `extend_maintenance`; ответ пользователю через callback `reply_fn`.
- **MAX-адаптер:** `adapters/max/` — polling через maxapi, обработчики: `/start`, `/help`, текстовые команды «текущие»/«список»/«сбои»/«работы», «остановить &lt;id&gt;», «продлить &lt;id&gt; &lt;минуты&gt;»; проверка `is_max_admin(user_id)`.
- **main.py:** при включённом MAX management запускается задача `run_max_polling(bot)` рядом с aiogram polling; при остановке бота задача отменяется.
- **Зависимости:** в `requirements.txt` добавлены `maxapi>=0.9.0`, `brotlicffi>=1.1.0`.
- **Сценарий «Сообщить» в MAX реализован:** команда «сообщить» запускает пошаговый диалог (сессии в `adapters/max/sessions.py`, логика в `adapters/max/create_flow.py`): выбор типа (1 — сбой, 2 — работа, 3 — обычное), ввод описания/текста, для сбоя — сервис, Jira (да/нет), при отсутствии Jira — SCM (да/нет), для работ — время начала/окончания и недоступные сервисы, затем Петлокал (да/нет) и подтверждение; финальное создание через `core.creation` (create_alarm, create_maintenance, send_regular_message).
