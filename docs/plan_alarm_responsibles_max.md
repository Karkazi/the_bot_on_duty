# План: привлечение ответственных в чат сбоя (MAX)

Документ описывает, как реализовать автоматическое упоминание ответственного и его руководителя в чате сбоя (FA-чат) на основе таблицы соответствия сервисов в Confluence.

---

## 1) Цель и ожидаемое поведение

При создании сбоя бот:

1. Определяет сервис сбоя.
2. Находит в Confluence ответственного и руководителя по этому сервису.
3. Отправляет в FA-чат отдельное сообщение с упоминаниями через `@`.

Обязательные участники для первой версии:

- Королев Максим Олегович
- Редькин Александр Сергеевич

---

## 2) Текущее место интеграции

Наиболее безопасная точка в текущем коде — после успешного создания/выбора FA-чата в `core/creation.py` (рядом с отправкой первого сообщения о сбое в чат).

Почему здесь:

- уже известны `alarm_id`, `service`, `max_chat_id`;
- уже выполнена основная логика создания сбоя;
- можно сделать best-effort (ошибка упоминания не ломает создание сбоя).

---

## 3) Источник данных: таблица Confluence

Используем страницу Confluence для теста/старта:

- <https://confluence.petrovich.tech/pages/viewpage.action?pageId=304990042>

Выбран **вариант A**: mention берём напрямую из таблицы Confluence.

Источник значений:

- `Ответсвенный за устранение сбоя` — кого тегаем для устранения;
- `Ответственный за сервис/систему` — кого информируем о сбое;
- `Имя автора сообщения` — строка для mention в MAX.

Правило формирования mention для MAX:

- если в строке Confluence `Имя автора сообщения: Aleredkenen`, то в MAX используем `@Aleredkenen`.
- то есть бот формирует mention как `@` + значение поля `Имя автора сообщения` без пробелов по краям.

Плюсы выбранного варианта:

- минимальные риски внедрения;
- нет зависимости от отдельного API поиска пользователей MAX.

Ограничение:

- корректность упоминания зависит от заполнения поля `Имя автора сообщения` в Confluence.

---

## 4) Предлагаемая архитектура

### 4.1 Новый сервис

`services/responsibles_service.py`:

- загрузка таблицы Confluence;
- нормализация названий сервисов;
- кэш в памяти (TTL, например 10-15 минут);
- метод:
  - `get_responsibles_for_service(service_name) -> {responsible_mention, manager_mention, source_row}`

### 4.2 Нормализация сервиса

Нужен единый helper:

- trim;
- lower-case;
- словарь синонимов (например `мах`/`max`, сервисные алиасы).

Кейс `Другое`:

- если выбран `Другое: ...`, пытаться искать по уточнению;
- если не найдено — fallback-сообщение без @ (или с уведомлением дежурному о ручном добавлении).

### 4.3 Отправка в чат сбоя

В `core/creation.py` после первого сообщения в FA-чат:

- вызвать сервис поиска ответственных;
- сформировать и отправить отдельное сообщение в этот же `max_chat_id` через `MaxService.send_message(...)`.
- отправка выполняется только в FA-чате (в остальных каналах поведение не меняется).

Ошибки:

- логировать предупреждение;
- не прерывать сценарий создания сбоя;
- если по сервису ошибка в Confluence-таблице (например, пустое/некорректное поле `Имя автора сообщения`), в FA-чат отправлять сообщение в старом виде (без `@`), а инициатору отправлять сообщение:
  - `У указаного сервиса в таблице по ссылке ошибка в Ответвенных: <описание ошибки>`.

## 5) Формат сообщения (согласованный)

Используем **вариант 1**:

```text
Добрый день.
@ОтветственныйЗаУстранениеСбоя, вы назначены ответственным за данный сбой. Просьба оперативно подключиться к решению.
@РуководительОтветственного, добавляем вас для информации. В случае недоступности ответственного просьба назначить замену.
```

## 6) Конфигурация

Новые переменные `.env` (предложение):

- `RESPONSIBLES_ENABLED=1`
- `RESPONSIBLES_CONFLUENCE_PAGE_ID=<pageId>` (или label-поиск)
- `RESPONSIBLES_CACHE_TTL_SEC=900`
- `RESPONSIBLES_NOTIFY_ON_MISS=1` (лог/уведомление админу, если сервис не найден)

---

## 7) Наблюдаемость и контроль

Логи:

- найдено соответствие: сервис -> ответственный/руководитель;
- не найдено соответствие;
- ошибка чтения Confluence;
- ошибка отправки mention в MAX.

Метрики (опционально):

- число сбоев с успешным mention;
- число fallback без mention.

## 8) Этапы внедрения

### Этап 1 (MVP)

- таблица Confluence (`pageId=304990042`) как источник ответственных;
- mention в MAX формируется из поля `Имя автора сообщения` (`@<имя>`);
- lookup + отправка в FA-чат;
- fallback без блокировки создания сбоя.

### Этап 2

- расширить правила нормализации/алиасы сервисов;
- уведомление дежурного при отсутствии маппинга.

### Этап 3

- автоматический маппинг ФИО/email -> MAX mention (если появится устойчивый источник данных).

---

## 9) Тест-план

1. Сервис найден в таблице:
   - создаём сбой -> в FA-чат уходит сообщение с двумя `@`.
2. Сервис не найден:
   - сбой создаётся;
   - бот логирует отсутствие соответствия;
   - в FA-чат отправляется fallback-текст без `@`;
   - инициатору отправляется сообщение о проблеме в Confluence-таблице с описанием ошибки.
3. Confluence недоступен:
   - сбой создаётся;
   - ошибка не прерывает основной сценарий.
4. Кейс `Другое`:
   - проверяем поведение с уточнением и без.

---

## 10) Открытые вопросы для согласования

1. Нужен ли пинг только при создании сбоя, или и при продлении/эскалации.
   - Решение: пинг отправляется только при создании сбоя.
   - При продлении/эскалации повторные пинги не отправляются.
