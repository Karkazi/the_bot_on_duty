## Этап 1. Календарь регламентных работ (прод)

### 1.1. Ежедневная сводка в 10:00 и 18:00

**Требование:** бот присылает дежурному **простым списком** все работы, у которых начало или окончание попадают в **текущие календарные сутки** (00:00–23:59 по `BOT_TIMEZONE`), либо работа **пересекает** эти сутки (уточнить на приёмке).

**Получатели:** `MAX_CALENDAR_ADMIN_IDS` (как для запросов по календарю).

**Пример сообщения:**

```
📅 Регламентные работы на 15.05.2026

1. Обновление ЗУП 3.1
   05.04.2026 21:00 — 05.04.2026 23:59 | ЗУП | Просветова А.
   Информирование: 05.04.2026 18:00

2. …
```

**Задачи разработки:**

1. Вынести форматирование списка в `services/confluence_service.py` или отдельный `services/calendar_digest_service.py`.
2. Фоновая задача `send_daily_maintenance_digest` в `main.py`:
   - cron-подобная логика: раз в минуту проверять «сейчас 10:00 или 18:00» и что за сегодня digest ещё не отправлялся (флаг в `bot_state` или `data/digest_state.json`);
   - загрузка страниц через `get_confluence_calendar_page_ids()` + `parse_works_table()`;
   - фильтр по дате на «сегодня»;
   - отправка в MAX каждому из `MAX_CALENDAR_ADMIN_IDS`.
3. Переменные `.env` (опционально):
   - `CALENDAR_DIGEST_TIMES=10:00,18:00`
   - `CALENDAR_DIGEST_ENABLED=1`

**Критерии приёмки:**

- [ ] В 10:00 и 18:00 дежурный получает один список на сутки.
- [ ] Повторная отправка в тот же слот в тот же день не дублируется.
- [ ] Пустой список: короткое сообщение «На сегодня работ в календаре нет».

---

### 1.2. Вопрос об информировании (уже есть — довести до прода)

**Требование:** в колонке «Дата и время информирования» наступило время → дежурному уходит запрос с кнопками.

**Текущая логика:** опрос каждые ~10 с, отправка при `now >= inform_at`.

**Задачи перед продом:**

1. Проверить на боевой странице Confluence (label / `pageId`).
2. Убедиться, что `MAX_CALENDAR_ADMIN_IDS`, Confluence и MAX настроены в `.env` на сервере.
3. Прогнать сценарий: новая строка → ожидание `inform_at` → «Информировать» / «Не информировать».

**Критерии приёмки:**

- [ ] Запрос приходит не раньше `inform_at` (допуск до ~10 с).
- [ ] Работы с пустой колонкой «Оповещения» не беспокоят дежурного (если так зафиксировано в регламенте).

---

### 1.3. Информирование всегда MAX + Петлокал

**Решение встречи:** при подтверждении «Информировать» каналы **не читать** из колонки «Информирование бизнеса», а **всегда**:
- `send_to_max = True`
- `publish_petlocal = True`
- `send_to_telegram = False` (если Telegram отключён — без изменений)

**Задачи:**

1. Упростить ветку `conf_notify_` в `adapters/max/handlers.py` (убрать разбор «Петлокал / МАХ / цифры»).
2. При ручном добавлении работы в календарь через бота — согласовать те же правила в `cal_notify_*` (или оставить выбор только для ручного ввода — **уточнить у заказчика**).
3. Обновить `docs/maintenance_auto_notifications.md`.

**Критерии приёмки:**

- [ ] После «Информировать» из календаря уходит пост в MAX-канал и на Петлокал.
- [ ] Текст работы на Петлокале совпадает с форматом ручного заведения работ.

### 2.2. Статистика по сбоям и сервисам за период

**Требование:** любой участник процесса может **запросить у бота** отчёт:
- какие сервисы «болели» и сколько раз;
- что именно указывали в «Другое» за период.

**Предлагаемый UX (MAX / Telegram):**

- Команда или кнопка: `📊 Статистика` / `/stats`.
- Шаги: период (сегодня / 7 дней / месяц / ввод дат) → выгрузка текстом или файлом.

**Источник данных:**

| Источник | Что даёт | Ограничение |
|----------|----------|-------------|
| `bot_state` + `data/state.json` | Только **активные** сбои | Истории нет |
| Jira (FA) | Закрытые и открытые задачи, custom field «Сервис» | Нужен доступ API, пагинация |
| Новый архив в `data/alarm_history.json` | Полный контроль | Дописать при `create_alarm` / `stop_alarm` |

**Рекомендация:** при закрытии сбоя дописывать запись в `data/alarm_history.json` (id, service, service_other_spec, created_at, closed_at, description) — для быстрых отчётов без нагрузки на Jira; опционально сверка с Jira.

**Задачи разработки:**

1. Модель записи истории сбоя + сохранение в `stop_alarm` / `create_alarm`.
2. Сервис `services/alarm_statistics_service.py`: агрегация по `service`, отдельная группировка `service_other_spec` при `service == Другое`.
3. Обработчики в MAX (`cmd_stats` или новая кнопка) и при необходимости в Telegram.
4. Права: `MAX_ADMIN_IDS` / `ADMIN_IDS` или отдельный `STATS_ALLOWED_IDS` в `.env`.

**Пример отчёта:**

```
📊 Сбои за 01.05.2026 — 15.05.2026

Всего: 12

По сервисам:
• Мобильное приложение — 4
• Jira + Confluence — 3
• Другое — 2
    — Шлагбаум (2)
    — СКУД офис (1)
…
```

**Критерии приёмки:**

- [ ] Отчёт за выбранный период приходит в личку инициатору в MAX.
- [ ] Блок «Другое» показывает все уточнения с количеством.
- [ ] Сбои, закрытые до внедрения архива, в отчёт не попадают (или помечается «данные с даты X»).

## Этап 3. Сбой: ответственные в чате MAX

**Требование:** бот добавляет в **чат сбоя** (FA) ответственных через @ и отправляет приветственное сообщение.

**Черновик текста (согласовать с заказчиком):**

> Добрый день.  
> @ОтветственныйЗаУстранениеСбоя, ты являешься ответственным за данный сбой. Прошу оперативно подключиться к решению.  
> @РуководительОтветственногоЗаУстранениеСбоя, добавляем тебя для информации (прошу следить и в случае недоступности ответственного подключить другого сотрудника).

**Открытые вопросы:**

1. Откуда брать @ → user_id в MAX (справочник в `.env`, Jira custom fields, Confluence, ручной ввод)?
2. Есть ли в MAX API: `add_chat_members` / invite по user_id?
3. Добавлять сразу после создания сбоя или по команде дежурного в чате?
4. Один шаблон на все сервисы или разные ответственные по `PROBLEM_SERVICES`?

**Задачи разработки:**

1. Исследование MAX API (в `services/max_service.py`).
2. Конфиг: `ALARM_RESPONSIBLE_MAP` или поля в Jira при создании FA.
3. Вызов после `_task_max_chat` в `create_alarm` или отдельный шаг после назначения FA-чата.
4. Логирование ошибок добавления (не блокировать создание сбоя).

**Критерии приёмки:**

- [ ] В чате FA появляются нужные участники.
- [ ] Сообщение с @ отображается корректно в MAX.
- [ ] При недоступности API сбой всё равно создаётся, дежурный видит предупреждение в логах/ответе.

## Этап 4. Время завершения сбоя в Jira (FA)

**Требование:** при закрытии сбоя в MAX бот проставляет в задаче **FA** фактическое **дату и время завершения** (поле Jira — уточнить id: resolution date / custom field «время окончания сбоя»).

**Текущее поведение:** `stop_alarm` → архив чата в Jira-комментарий (`process_max_chat_on_alarm_close`), SCM, Петлокал, каналы; **transition / поля времени** не вызываются.

**Задачи разработки:**

1. Уточить у владельцев Jira:
   - transition в статус «Сбой устранён» / `Failure Fixed`;
   - какое поле = «время завершения» (`customfield_…`).
2. Модуль `utils/jira_close_fa.py`: `close_failure_issue(issue_key, closed_at)` — transition + обновление поля.
3. Вызов из `core/actions.py` → `stop_alarm` при `has_jira` и успешном закрытии (параллельно с остальными задачами).
4. `closed_at = bot_now_naive()` в момент нажатия «Остановить» в MAX/TG.

**Критерии приёмки:**

- [ ] После закрытия в боте в Jira у FA видно корректное время завершения (±1 мин).
- [ ] Сбой без Jira (`has_jira=false`) — без ошибок.
- [ ] Ошибка Jira не мешает закрытию в боте (как для SCM/Петлокала).
